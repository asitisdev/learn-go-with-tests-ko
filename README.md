### 애플리케이션 구축

이제 *Go 기초* 섹션을 잘 소화했다면 Go 언어 기능의 대부분과 TDD를 수행하는 방법에 대한 탄탄한 기초가 있습니다.

다음 섹션에서는 애플리케이션 구축을 다룹니다.

각 챕터는 이전 챕터를 기반으로 하며, 제품 소유자가 지시하는 대로 애플리케이션의 기능을 확장합니다.

훌륭한 코드를 작성하는 데 도움이 되는 새로운 개념이 소개되지만 대부분의 새로운 자료는 Go의 표준 라이브러리에서 무엇을 달성할 수 있는지 배우는 것입니다.

이것이 끝나면 테스트로 뒷받침되는 Go로 애플리케이션을 반복적으로 작성하는 방법에 대해 강력한 이해를 갖게 될 것입니다.

* [HTTP 서버](http-server.md) - HTTP 요청을 수신하고 응답하는 애플리케이션을 만듭니다.
* [JSON, 라우팅 및 임베딩](json.md) - 엔드포인트가 JSON을 반환하도록 하고 라우팅하는 방법을 탐구합니다.
* [IO와 정렬](io.md) - 디스크에서 데이터를 유지하고 읽으며 데이터 정렬을 다룹니다.
* [커맨드 라인 & 프로젝트 구조](command-line.md) - 하나의 코드베이스에서 여러 애플리케이션을 지원하고 커맨드 라인에서 입력을 읽습니다.
* [시간](time.md) - `time` 패키지를 사용하여 활동 예약하기.
* [웹소켓](websockets.md) - WebSocket을 사용하는 서버를 작성하고 테스트하는 방법 배우기.

* [인수 테스트 확장하기](scaling-acceptance-tests.md) - 사소하지 않은 시스템에 대한 인수 테스트 작성의 복잡성을 관리하는 기술 배우기.
* [모킹, 스텁 및 스파이 없이 작업하기](working-without-mocks.md) - 페이크와 계약을 사용하여 더 현실적이고 유지 관리하기 쉬운 테스트를 만드는 방법에 대해 배우기.

### 질문과 답변

저는 종종 인터넷에서 다음과 같은 질문을 접합니다

> x, y, z를 수행하는 놀라운 함수를 어떻게 테스트합니까

그러한 질문이 있으면 github에 이슈로 올려주시면 문제를 해결하기 위한 짧은 챕터를 작성할 시간을 찾아보겠습니다. 이러한 콘텐츠는 테스트에 대한 사람들의 *실제* 질문을 다루기 때문에 가치 있다고 생각합니다.

* [OS exec](os-exec.md) - OS에 연락하여 데이터를 가져오는 명령을 실행하고 비즈니스 로직을 테스트 가능하게 유지하는 방법의 예.
* [에러 타입](error-types.md) - 테스트를 개선하고 코드를 더 쉽게 작업할 수 있도록 자체 에러 타입을 만드는 예.
* [Context-aware Reader](context-aware-reader.md) - 취소를 사용하여 `io.Reader` 증강을 TDD하는 방법 배우기. [Go용 Context-aware io.Reader](https://pace.dev/blog/2020/02/03/context-aware-ioreader-for-golang-by-mat-ryer) 기반
* [HTTP 핸들러 재방문](http-handlers-revisited.md) - HTTP 핸들러 테스트는 많은 개발자에게 존재의 고통인 것 같습니다. 이 챕터에서는 핸들러를 올바르게 설계하는 것과 관련된 문제를 탐구합니다.

## 배경

저는 개발 팀에 Go를 소개한 경험이 있으며 Go에 관심 있는 몇몇 사람들에서 매우 효과적인 Go 시스템 작성자로 팀을 성장시키는 방법에 대해 다양한 접근 방식을 시도했습니다.

### 효과가 없었던 것

#### *그* 책 읽기

시도한 접근 방식은 [블루북](https://www.amazon.co.uk/Programming-Language-Addison-Wesley-Professional-Computing/dp/0134190440)을 가져와 매주 연습 문제와 함께 다음 챕터를 논의하는 것이었습니다.

이 책을 좋아하지만 높은 수준의 헌신이 필요합니다. 책은 개념을 매우 상세하게 설명하는데 물론 훌륭하지만 진행이 느리고 꾸준합니다 - 이것은 모든 사람에게 적합하지 않습니다.

소수의 사람들은 X 챕터를 읽고 연습할 것이지만 많은 사람들은 그렇지 않았습니다.

#### 몇 가지 문제 해결

카타는 재미있지만 일반적으로 언어를 배우는 범위가 제한적입니다; 카타를 해결하기 위해 고루틴을 사용할 가능성이 낮습니다.

또 다른 문제는 열정의 수준이 다양할 때입니다. 일부 사람들은 다른 사람들보다 언어를 훨씬 더 많이 배우고 그들이 한 것을 시연할 때 다른 사람들이 익숙하지 않은 기능으로 사람들을 혼란스럽게 만듭니다.

결국 학습이 꽤 *비구조적*이고 *임시적*으로 느껴지게 됩니다.

### 효과가 있었던 것

지금까지 가장 효과적인 방법은 [go by example](https://gobyexample.com/)을 읽으면서 언어의 기초를 천천히 소개하고, 예제로 탐구하고 그룹으로 논의하는 것이었습니다. 이것은 "숙제로 x 챕터 읽기"보다 더 상호작용적인 접근 방식이었습니다.

시간이 지남에 따라 팀은 언어의 *문법*에 대한 탄탄한 기초를 얻었으므로 시스템 구축을 시작할 수 있었습니다.

이것은 기타를 배우려고 할 때 스케일을 연습하는 것과 유사해 보입니다.

자신이 얼마나 예술적이라고 생각하든 기초를 이해하고 역학을 연습하지 않으면 좋은 음악을 작성할 가능성이 낮습니다.

### 저에게 효과적인 것

*저*는 새 프로그래밍 언어를 배울 때 보통 REPL에서 놀면서 시작하지만 결국 더 많은 구조가 필요합니다.

제가 하고 싶은 것은 개념을 탐구한 다음 테스트로 아이디어를 굳히는 것입니다. 테스트는 작성한 코드가 올바른지 확인하고 배운 기능을 문서화합니다.

그룹과 함께 배운 경험과 제 개인적인 방식을 취하여 다른 팀에게 유용하게 증명될 수 있는 무언가를 만들려고 합니다. 작은 테스트를 작성하여 기초를 배우면 기존 소프트웨어 설계 기술을 사용하여 훌륭한 시스템을 출시할 수 있습니다.
