# 기여하기

기여를 환영합니다. 이 프로젝트가 테스트를 작성하면서 Go를 배우는 방법에 대한 훌륭한 가이드의 본거지가 되기를 바랍니다. PR을 제출하거나 이슈를 생성하는 것을 고려해 주세요. [여기](https://github.com/quii/learn-go-with-tests/issues)에서 할 수 있습니다.

## 찾고 있는 것

* Go 기능 가르치기 \(예: `if`, `select`, 구조체, 메서드 등\).
* 표준 라이브러리 내의 흥미로운 기능 소개. 예를 들어 HTTP 서버를 TDD하는 것이 얼마나 쉬운지 보여주세요.
* 벤치마킹, 레이스 감지기 등 Go의 도구가 훌륭한 소프트웨어에 도달하는 데 어떻게 도움이 되는지 보여주세요.

자신만의 가이드를 제출할 자신이 없다면, 배우고 싶은 것에 대한 이슈를 제출하는 것도 여전히 가치 있는 기여입니다.

### ⚠️ 새 콘텐츠에 대해 빠르게 피드백 받기 ⚠️

- TDD는 반복적으로 작업하고 피드백을 받도록 가르치며, 기여하고 싶다면 같은 방식을 권장합니다
    - 첫 번째 테스트와 구현으로 PR을 열고, 접근 방식을 논의하여 피드백과 방향 수정을 제공할 수 있도록 하세요
- 이것은 물론 오픈소스이지만 콘텐츠에 대한 강한 의견이 있습니다. 빨리 대화할수록 좋습니다.

## 스타일 가이드

* 항상 TDD 사이클을 강화하세요. [챕터 템플릿](template.md)을 참조하세요.
* 테스트에 의해 주도되는 기능에 대한 반복을 강조하세요. Hello, world 예제는 점진적으로 더 정교해지고 테스트에 _의해_ 새로운 기법을 배우기 때문에 잘 작동합니다. 예를 들어:
  * `Hello()` \<- 함수 작성 방법, 반환 타입.
  * `Hello(name string)` \<- 인수, 상수.
  * `Hello(name string)` \<- `if`를 사용하여 "world"로 기본값 설정.
  * `Hello(name, language string)` \<- `switch`.
* 필요한 지식의 표면적을 최소화하려고 노력하세요.
  * 독자를 다른 기능으로 혼란스럽게 하지 않으면서 가르치려는 것을 보여주는 예제를 생각하는 것이 중요합니다.
  * 예를 들어 포인터를 이해하지 않고도 `struct`에 대해 배울 수 있습니다.
  * 간결함이 왕입니다.
* [Code Review Comments 스타일 가이드](https://go.dev/wiki/CodeReviewComments)를 따르세요. 모든 섹션에서 일관된 스타일을 유지하는 것이 중요합니다.
* 섹션 끝에는 실행 가능한 애플리케이션이 있어야 합니다 \(예: `main` 함수가 있는 `package main`\) 사용자가 실제로 작동하는 것을 보고 놀 수 있도록 합니다.
* 모든 테스트가 통과해야 합니다.
* PR을 올리기 전에 `./build.sh`를 실행하세요.
